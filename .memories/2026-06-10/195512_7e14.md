## ES-DE launcher-metadata writer (ROM management)

Built the first launcher-metadata writer so transferred ROMs show clean display names instead of canonical No-Intro filenames.

### What
- New `server/utils/launcherMetadata.ts` — `writeLauncherMetadata(kind, destRoot, entries)` + `launcherKindForCacheKey(cfg, cacheKey)`. ES-DE writer upserts `<game><path>./file</path><name>Display</name></game>` into `<destRoot>/<systemKey>/gamelist.xml`, merging via fast-xml-parser (preserves existing scraped `<desc>`/`<rating>`/attributes; only touches our games' `<name>`). Atomic tmp+rename. muOS returns a "not implemented yet" per-system result.
- `LauncherKind = "es-de" | "muos"` + optional `launcherKind` on `DeviceIdentity` and `VirtualMount` (`types.ts`). Optional → no config migration.
- Transfer hook: `transfer/execute.post.ts` writes metadata for successfully-copied games after copy, before rescan; returns `metadata` in response.
- Standalone `server/api/roms/metadata/sync.post.ts` — pushes names for already-installed games without copying ROMs (for name edits after transfer).
- PATCH endpoints (`devices/[id]`, `virtual-mounts`) accept/validate `launcherKind`.
- UI: launcher picker on device page ROM-library card; "Sync names to device" button + metadata result display on transfer page.

### Why
Master keeps canonical filenames for CRC/scan stability (design decision 3). Renaming on copy was rejected. gamelist.xml is ES-DE's name source, so writing it is the only non-renaming way to clean names = the feature's differentiator (decision 6).

### Key discoveries (corrected the brief's guesses)
- ES-DE **ignores** in-rom-tree gamelist.xml by default; user enabled `LegacyGamelistFileLocation` so it reads/writes `<romsRoot>/<system>/gamelist.xml` — exactly the destRoot this app scans. No separate ES-DE-home path needed.
- `<path>` needs leading `./`, relative to system folder. ES-DE matches media by filename, not a tag.
- muOS is NOT SQLite (brief was wrong). Names via `MUOS/info/name/content.json` + per-catalogue files; `catalogue/<sys>/text/*.txt` is a description blurb, not the list name. Format still unconfirmed → deferred.
- `LegacyGamelistFileLocation` relocates only the gamelist, NOT media — boxart stays in ES-DE internal storage (separate later feature).
