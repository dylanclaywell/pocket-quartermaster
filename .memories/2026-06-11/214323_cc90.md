## muOS name writer (Step 1 of muOS support)

muOS DOES support general per-game names (confirmed from source, not just docs — the user was right).

### Source proof (MustardOS/frontend, muxshare.c: resolve_friendly_name)
- Reads `MUOS/info/name/<content-dir>.json`, falls back to `MUOS/info/name/global.json`.
- JSON object, key = **lowercased ROM filename without extension**, value = display name.
- Read DIRECTLY at runtime (no `mulookup --gen` needed). No setting required (the "Friendly Folder Names" toggle is only for folder.json = folder names).
- Docs say `content.json`; source uses `global.json` — trust source.

### Built
- `muosRootRelPath` per-device/VM config (path to the MUOS folder containing info/). SourceRoot carries it; both PATCH endpoints validate it.
- `metadataTargetRoot` muos branch → `<mount>/<muosRootRelPath>/info`.
- `writeMuos(infoRoot, entries)` in launcherMetadata.ts: merges into `<info>/name/global.json`, key=lower(stem) → displayName, preserves existing keys, atomic write. Returns one MetadataWriteResult (systemKey "names"). Replaced the old "not implemented" muOS stub.
- Chose global.json (folder-agnostic fallback) over per-content-dir json so we don't have to match the device's exact folder names; rare cross-system stem collisions share an identical display name anyway.
- Device page: muOS section (gated launcherKind==='muos') — muOS folder picker + "Sync names to launcher" button (reuses the existing device-scoped syncNames → /api/roms/metadata/sync, which now handles muos) + note.
- The transfer-time name write + the standalone sync both now work for muOS via the existing generic flow.

### Verified
- typecheck clean; devices + transfer pages 200.
- Merge simulation: "Pokemon Emerald (USA) (Rev 1).gba" → key "pokemon emerald (usa) (rev 1)" = "Pokemon Emerald"; existing entries preserved. Matches resolve_friendly_name exactly.
- True device write = user hardware test.
