## DIRECTION CHANGE: ROM management → multi-source, any-to-any (drops the master-library model)

User explicitly pivoted the ROM-management architecture. This SUPERSEDES the active brief's decision #1 ("concrete single-master schema") and adopts the brief's previously-"rejected alternative" (abstract/peer model).

### New model
- **No master library.** The server catalogs games from EVERY configured ROM source, not one designated library. The catalog = union of games across all devices/mounts.
- **Each game tracks which devices hold which variant.** Library/browse views become "here's what the selected device has" (or All sources).
- **Transfer is any source → any destination, any direction** (peer-to-peer), not library→destination.
- **The server is the full metadata + art catalog** for every game present on any device.
- `romLibrarySourceKey` + the library/destination role split become obsolete.

### Why this is now OK (it was deferred before)
User: performance is not a near-term concern — small library, already cataloged across all devices by swapping them in, and "if the Pi Zero 2 W isn't powerful enough at launch I'll pick another device or put it on my main server." So the Pi-Zero hard constraint is softened to a preference (updated memory/project_deploy_target_pi_zero_2w.md).

### Implementation impact (planned, not yet built)
1. computeLibrary: fold ALL cached sources into games/variants; each variant tracks which sources hold it; per-(game,source) presence matrix. Remove library/destination roles + effectiveLibraryKey.
2. /roms: replace master-library picker with a source/device selector ("Showing: All | Device A | …").
3. Transfer: Source dropdown (any source) + Destination dropdown (any other); buildTransferPlan resolves the variant from the chosen SOURCE.
4. Art push (original Phase 2) slots after — art source is the server cache, unaffected by any-to-any.
